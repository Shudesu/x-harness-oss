# X Harness 運用セキュリティガイド

X Harness は、投稿、返信、DM、フォローなど、X アカウントに直接影響する操作を
管理画面、API、MCP、Cron から実行できます。本番運用では「管理画面へログインできる」
ことだけを安全の根拠にせず、認証情報、公開範囲、自動実行を一体として管理してください。

このガイドは運用上の注意点をまとめたものです。デプロイ先の構成、X Developer
Platform の最新の利用規約、適用される法令や組織のセキュリティ基準も別途確認してください。

## 最初に理解する認証情報

| 情報 | 用途 | 漏えい時の主な影響 |
|---|---|---|
| `API_KEY` / `X_HARNESS_API_KEY` | Worker API と管理画面の管理認証 | X Harness 上の高権限操作を実行される可能性 |
| Consumer Key / Secret | X アプリの識別と署名 | Access Token と組み合わせた X API 操作 |
| Access Token / Secret | 登録した X ユーザーとしての操作 | 許可された範囲での投稿、DM、フォロー等 |
| `TWITTER_AUTH_TOKEN` / `TWITTER_CT0` | `twitter-cli` の Cookie 認証 | ブラウザセッション相当の不正利用 |

`API_KEY` は X Developer Platform が発行するキーではありません。X Harness の
管理 API キーであり、環境変数に設定したキーは管理者権限として扱われます。
十分な長さのランダム値を生成し、パスワードマネージャーまたはシークレットストアで
保管してください。ソースコード、Issue、チャット、シェル履歴、CI ログへ記録しないでください。

環境ごとにすべてのキーを分離し、担当者の離任、端末紛失、誤公開、不審な操作があった場合は
関連するキーとトークンを失効・再発行してください。

## 最小権限で構成する

README のセットアップ例は全機能を利用するため `Read, Write, and Direct Messages`
を案内しています。DM を使わない運用では、X アプリに DM 権限を付けないでください。
読み取り専用の収集と投稿運用も、可能なら別のアプリまたは別のアカウントに分離します。

X Harness のスタッフ用キーは、担当業務に必要な最小ロールを割り当てます。環境変数の
`API_KEY` は管理者権限を持つため、日常作業で共有せず、個別のスタッフ用キーを使用してください。

MCP に設定する `X_HARNESS_API_KEY` も同じ権限を引き継ぎます。AI クライアントへ接続する前に、
利用可能なツールと対象アカウントを確認し、書込み操作には人間の確認を挟んでください。

## 公開範囲を確認する

Worker の認証ミドルウェアには、ヘルスチェック、セットアップ、トークン取得、画像配信、
ユーザー検索、ゲート検証など、認証なしで利用することを想定したルートがあります。
デプロイ前に `apps/worker/src/middleware/auth.ts` の公開ルートを確認し、自分の運用で
公開が必要なものだけがインターネットから到達できる構成にしてください。

管理 API と管理画面は、可能なら Cloudflare Access、IP 制限、専用ホスト名などで追加保護します。
ただし webhook やキャンペーン検証など、外部からのアクセスが必要なルートまで一律に遮断すると
機能しません。パス単位のポリシー、または公開 API と管理 API のホスト名分離を検討してください。

CORS はブラウザからの呼び出し元を制御する仕組みであり、API の認証にはなりません。
`workers.dev` の既定 URL を残す場合も、カスタムドメイン側の保護を迂回できないか確認してください。

## Cookie 認証を隔離する

`twitter-cli` 用の Cookie は、通常の API トークンと同様に機密情報です。収集専用の
サブアカウントを使用し、本番投稿アカウントのブラウザ Cookie は使用しないでください。
Cookie はローカルの MCP プロセスだけに渡し、Worker、D1、リポジトリ、CI へ送信しないでください。

Cookie を利用する収集方法については、X の最新の利用規約と組織のポリシーを確認してください。
アカウント制限や仕様変更を前提に、停止しても投稿運用へ波及しない構成にします。

## 自動実行を段階的に有効化する

投稿やDMなどの外部操作は、実行後に完全には取り消せません。次の順序で有効化してください。

1. 開発、ステージング、本番で Cloudflare リソース、X アカウント、認証情報を分離する
2. 専用のステージング用 X アカウントで、読取り操作から確認する
3. 投稿本文、宛先、実行時刻、対象アカウントを人間が確認して単発の書込みを試す
4. 失敗時の停止担当者と、Cron、Worker、X トークンを止める手順を確認する
5. 監視と利用上限を設定した後で、必要な自動実行だけを有効化する

`--dangerously-skip-permissions` を付けた AI クライアントや無人 Cron は、確認なしに
外部操作へ到達する可能性があります。専用の低権限環境でのみ使用し、入力データや取得した
Web コンテンツを命令として扱わないようにしてください。

## データと復旧手順を準備する

D1 のスキーマ変更や本番デプロイの前にバックアップを取得し、復元手順を実際に確認します。
本番とステージングで D1、R2、Worker、Pages の名前と ID を明確に分け、デプロイ対象を
コマンド実行前に再確認してください。

ログには API キー、X トークン、Cookie、DM 本文などを出力しないでください。運用に必要な
監査情報は、誰が、いつ、どのアカウントへ、どの種類の操作を実行したかに限定し、保持期間と
閲覧権限を定めます。

インシデント時に迷わないよう、少なくとも次の順序を手順化してください。

1. Cron と自動実行クライアントを停止する
2. Worker の管理アクセスを遮断する
3. X の Access Token と関連する管理 API キーを失効する
4. D1 の操作履歴、Cloudflare のログ、X 側の履歴を保全する
5. 影響範囲を確認してから、キーを再発行し段階的に復旧する

## 本番公開チェックリスト

- [ ] 本番、ステージング、開発のアカウントとリソースが分離されている
- [ ] 管理 `API_KEY` はランダム生成され、安全な保管先にだけ保存されている
- [ ] 日常運用者には個別キーと最小ロールが割り当てられている
- [ ] X アプリの Write / DM 権限は実際に使う機能だけに限定されている
- [ ] 認証なしで到達できるルートと、その公開理由を確認した
- [ ] 管理 API と管理画面に追加のアクセス制御がある
- [ ] `workers.dev` URL を含め、アクセス制御の迂回経路がない
- [ ] MCP と Cron の書込み操作に承認または停止手段がある
- [ ] 専用ステージングアカウントで単発の書込みを確認した
- [ ] D1 のバックアップと復元を確認した
- [ ] X API の利用量、Cloudflare エラー、不審な操作を監視できる
- [ ] キー失効と自動実行停止の手順・担当者が決まっている
